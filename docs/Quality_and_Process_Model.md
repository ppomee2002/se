# 프로세스 모델 선정 및 품질 관리 계획

## 1. 개발 대상 시스템 특성 정리
* **기존 사례 존재**  
Google Drive, OneDrive 등 벤치마킹이 가능한 성숙한 유사 시스템이 존재
* **보안 복잡성**  
사내 주요 자산인 파일을 다루므로 전송 암호화, 권한 제어 등 높은 수준의 기술적 보안 구현이 필요
* **규모**  
약 200명 규모의 사내 시스템이나, 데이터 안정성 및 무결성에 대한 요구치는 높은 편이다
* **확장성**  
초기 핵심 기능 구축 이후 향후 AI 기반 문서 검색 및 실시간 공동 편집 기능 등 신기술 적용 여지가 높음

## 2. 적용 가능한 프로세스 모델 선정 - 나선형 모델 (Spiral Model)

### 2.1 선정 이유 및 전략
본 프로젝트는 단순한 기능 구현보다 사내 데이터 보호와 시스템 안정성이 최우선이므로, 각 단계마다 반복적인 위험 분석을 수행하여 잠재적 손실을 최소화하는 나선형 모델이 가장 적합하다.

### 2.2 구체적인 위험 요소(Risk) 예시 및 대응 방안
나선형 모델의 각 회전(Iteration) 주기에서 위험 요소를 집중 점검
* **데이터 무결성 위험**  
네트워크 불안정으로 인한 파일 업로드 중 손상 위험 -> (대응) 반복 주기 내 체크섬(Checksum) 검증 로직 구현 및 테스트
* **권한 관리 로직 오류**  
외부 공유 링크를 통한 비인가자의 데이터 접근 위험 -> (대응) 프로토타입 단계에서 시나리오 기반 권한 침투 테스트 실시
* **성능 병목 위험**  
동시 접속자 집중 시 검색 엔진 및 DB 부하 발생 위험 -> (대응) 각 주기마다 부하 테스트(JMeter 등)를 수행하여 인프라 확장 계획 수립

## 3. 핵심 품질 요소 중심의 관리 계획
| 품질 요소 | 관리 목표 | 측정 및 관리 방법 |
| :--- | :--- | :--- |
| **신뢰성** | 데이터 손실율 0.0% 달성 | 결함 허용(Fault Tolerance) 테스트 및 정기적인 백업/복구 시나리오 검증 |
| **보안성** | 비인가 접근 및 유출 차단 | 권한별 리소스 접근 제어 리스트(ACL) 검증 및 전송 구간 암호화 점검 |
| **성능** | 명확한 응답 지표 준수 | JMeter 등을 활용한 동시 접속 부하 테스트 및 응답 시간(Latency) 모니터링 |
| **유지보수성** | 변경 용이성 및 가독성 확보 | 정적 코드 분석 도구 활용 및 팀 내 교차 코드 리뷰 의무화 |
