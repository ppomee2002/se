# 미니 드라이브 프로젝트 관리 계획서

## 1. 프로젝트 개요

본 프로젝트는 조직 내부에서 사용할 수 있는 클라우드 파일 공유 시스템인 **미니 드라이브**를 개발하는 것을 목표로 한다.

현재 파일 공유가 이메일, 메신저, 개인 클라우드 등에 흩어져 이루어지면 최신 파일 확인이 어렵고, 파일 검색에 시간이 오래 걸리며, 외부 공유 시 보안 문제가 발생할 수 있다.

미니 드라이브는 이러한 문제를 해결하기 위해 파일을 중앙에서 저장하고, 사용자들이 쉽게 업로드·다운로드·공유·검색할 수 있도록 하는 웹 기반 시스템이다.

---

## 2. 프로젝트 목표

본 프로젝트의 주요 목표는 다음과 같다.

- 조직 내 파일을 중앙에서 관리할 수 있도록 한다.
- 사용자가 파일을 쉽게 업로드하고 다운로드할 수 있도록 한다.
- 폴더 기능을 통해 파일을 체계적으로 관리할 수 있도록 한다.
- 사용자 간 파일 공유 기능을 제공한다.
- 외부 사용자와 공유할 수 있는 링크 기능을 제공한다.
- 파일 검색 기능을 통해 필요한 자료를 빠르게 찾을 수 있도록 한다.
- 파일 수정 시 이전 버전을 확인하고 복원할 수 있도록 한다.
- 로그인과 권한 관리를 통해 파일 접근 보안을 강화한다.

---

## 3. 주요 기능

| 구분 | 기능 | 설명 |
|---|---|---|
| F01 | 파일 업로드 | 사용자가 파일을 시스템에 업로드한다. |
| F02 | 파일 다운로드 | 사용자가 저장된 파일을 다운로드한다. |
| F03 | 폴더 관리 | 폴더 생성, 이름 변경, 삭제, 파일 이동을 지원한다. |
| F04 | 파일 공유 | 다른 사용자에게 파일을 공유한다. |
| F05 | 공유 권한 설정 | 보기, 수정, 댓글 권한을 설정한다. |
| F06 | 외부 링크 공유 | 외부 협력 업체와 공유할 수 있는 링크를 생성한다. |
| F07 | 파일 검색 | 파일 이름, 업로드 날짜, 사용자, 파일 유형으로 검색한다. |
| F08 | 버전 관리 | 이전 버전 확인, 다운로드, 복원을 지원한다. |
| F09 | 사용자 계정 | 이메일과 비밀번호를 이용해 로그인한다. |
| F10 | 보안 관리 | 인증, 권한 기반 접근 제어, 데이터 전송 암호화를 적용한다. |

## 4. 개발 방식

본 프로젝트는 **애자일 기반의 스크럼 방식**으로 진행한다.

처음부터 모든 기능을 완성하는 방식이 아니라, 기능을 작은 단위로 나누어 순차적으로 구현하고, 각 단계마다 리뷰와 피드백을 반영한다.

진행 방식은 다음과 같다.

1. 고객 요구사항을 분석한다.
2. 필요한 기능을 목록으로 정리한다.
3. 우선순위가 높은 기능부터 개발한다.
4. 기능 구현 후 테스트를 수행한다.
5. GitHub issue 또는 pull request를 통해 리뷰를 받는다.
6. 리뷰 내용을 반영하여 문서를 수정한다.

---

## 5. 작업 분해 구조

프로젝트는 다음과 같은 흐름으로 진행한다.

- **요구사항 분석**
  - 고객 요구사항 문서를 검토하고 필요한 기능을 정리한다.
  - 파일 업로드, 다운로드, 공유, 검색, 버전 관리 기능을 중심으로 요구사항을 정리한다.
  - 보안, 성능, 사용성과 같은 비기능 요구사항도 함께 확인한다.

- **설계**
  - 사용자가 파일을 관리하는 화면 흐름을 설계한다.
  - 사용자, 파일, 폴더, 권한 정보를 저장할 데이터베이스 구조를 설계한다.
  - 업로드된 파일의 저장 방식과 관리 방식을 정리한다.

- **구현**
  - 로그인, 파일 업로드/다운로드, 폴더 관리 기능을 구현한다.
  - 파일 공유, 검색, 버전 관리 기능을 구현한다.
  - 기능별 구현이 끝나면 GitHub에 작업 내용을 올린다.

- **테스트**
  - 각 기능이 요구사항에 맞게 동작하는지 확인한다.
  - 공유 권한과 접근 제한이 올바르게 적용되는지 테스트한다.
  - 발견된 오류는 수정 후 다시 확인한다.

- **문서화**
  - 프로젝트 관리 계획서, 요구사항 정리 문서, README를 작성한다.
  - 조원 또는 다른 팀원의 리뷰를 반영하여 문서를 업데이트한다.

## 8. 품질 관리

프로젝트 품질을 높이기 위해 다음과 같은 방식을 사용한다.

* GitHub를 이용하여 코드와 문서를 관리한다.
* 기능 추가나 수정 사항은 issue로 기록한다.
* 다른 팀원의 문서는 issue 또는 pull request로 리뷰한다.
* 리뷰를 받은 내용은 문서에 반영하고 다시 GitHub에 업데이트한다.
* 주요 기능은 구현 후 직접 테스트한다.
* 파일 공유와 권한 관리는 보안 문제가 생길 수 있으므로 별도로 확인한다.

## 10. 리스크 관리

| 위험 요소    | 설명                            | 대응 방안                         |
| -------- | ----------------------------- | ----------------------------- |
| 요구사항 변경  | 개발 중 필요한 기능이 바뀔 수 있음          | issue에 기록하고 우선순위를 다시 정한다.     |

## 11. GitHub 리뷰 및 반영 계획

과제 요구에 따라 다른 조원이 작성한 문서를 확인하고 GitHub issue 또는 pull request를 통해 리뷰를 남긴다.

리뷰할 때는 다음 항목을 중심으로 확인한다.

* 프로젝트 목표가 명확한가
* 개발 일정이 적절한가
* 주요 기능이 빠지지 않았는가
* 역할 분담이 자연스러운가
* 리스크 대응 방안이 구체적인가

다른 사람이 남긴 리뷰는 확인 후 필요한 내용을 문서에 반영한다. 수정한 뒤에는 commit message에 어떤 내용을 수정했는지 기록하고 GitHub에 다시 업로드한다.
